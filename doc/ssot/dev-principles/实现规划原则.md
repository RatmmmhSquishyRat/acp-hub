# 实现规划原则

1. 在实现任何非小型重构, 非小型模块的情况下, 都需要根据相关的ssot&pillars, 给出五个文档:
   1. spec文档和design文档
   2. BDD文档和TDD文档
   3. impl_plan文档
   而这些文档, 在相关ssot&pillar变更时候, 都要进行完整检查和重构更新.
2. 对于这个实现者, 也要单独给出role文档, 自己的对外工作职责介绍, 工作原则, BOOTSTRAP等等文档, 并和上述文档要求一致.
3. 并且, 给出文档之后, 不能够直接开始实现, 而是要使用第三方视角批判性的subagents, 从多种distinct初始视角, 根据相关ssot&pillars对于相关文档进行完整review和对抗性辩论, 并进行review-rework loop, 直到所有文档, 完全符合相关ssot&pillars, 在这个指标上完全闭合. 之后才能够考虑开始实现.

4. 用户于 2026-07-18 新增的持续维护原则（以下原文保留）：

   > 我看到你的hub文件已经1000行以上了, 及时自行规划设计, 完整进行文件和代码拆分, 不要等我来告诉你

   该原则适用于实现者发现单文件职责持续扩张的情形。实现者需要主动检查模块边界，形成设计、验收与实施计划，并完成拆分和第三方评审；不能把发现和决策责任退回给用户。
